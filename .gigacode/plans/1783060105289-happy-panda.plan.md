## Обновлено 06.07.2026 (обновление с исправлением индексов)

## Основные требования
1. Заменить input field на выпадающее меню (DropdownMenu) в AdministratorScreen для выбора количества охранников
2. Dropdown должен иметь три позиции:
   - "1 охранник (старший)" (1 guard - shift leader)
   - "Старший и 1 охранник" (Shift leader + 1 guard)
   - "Старший и 2 охранника" (Shift leader + 2 guards)
3. NFC-идентификация при начале обходов
4. Для групповых смен must выбрать старшего смены с ролью "старший смены"
5. Кнопка "ОТКРЫТЬ СМЕНУ" на экране приветствия при guardsCount > 1 и закрытой смене
6. Для guardsCount = 1: NFC-сканирование напрямую запускает смену (без кнопки)
7. В RoundsScreen при guardsCount > 1 показывать диалог выбора охранника при нажатии "Начать обход"
8. При guardsCount > 1 и выходе из OhrannikCabinetScreen во время обхода - кнопка "Продолжить обход" красным
9. При guardsCount > 1 и нажатии "Назад" в OhrannikCabinetScreen - возврат напрямую в rounds
10. TopAppBar в OhrannikCabinetScreen показывает имя выбранного охранника (из диалога)
11. Кнопка "Продолжить обход" красным - только для реально начатых обходов (checkpointIndex > 0)
12. Кнопка "Начать обход" синим - для незапущенных обходов

## Технические детали

### Архитектура работы

**Режим 1: Стандартный (guardsCount = 1)**
1. На экране приветствия текст "Поднесите личную карту к телефону"
2. При NFC-сканировании сразу начинается смена

**Режим 2: Групповой (guardsCount > 1)**
1. На экране приветствия кнопка "ОТКРЫТЬ СМЕНУ" (если смена закрыта)
2. Нажатие → экран "GuardNfcSelectionScreen"
3. Автоматическое NFC-сканирование (без кнопки)
4. Сканирование старшего смены → охранники
5. После выбора всех → кнопка "НАЧАТЬ СМЕНУ"
6. Смена начинается, переход на экран "rounds"
7. В экране "rounds" показывается список всех охранников

### Обратная совместимость
- guardsCount = 1 работает как и раньше
- Старый экран guard_selection (через список) работает параллельно
- База данных ShiftRecord обновлена для хранения guardList
- Все отчеты и логи корректно работают с группами
- Per-round checkpoint index (`round_${roundIndex}_checkpoint_index`) для корректного отслеживания прогресса каждого обхода

# План реализации системы передачи отчетов между устройствами

## Обновлено 07.07.2026

## Вопрос и ответ

**Вопрос:** Возможно ли создать отдельное приложение на Android для приема сообщений и удаленной настройки приложения без использования сторонних сервисов?

**Ответ:** Да, возможно, но с ограничениями:

### Варианты организации связи:

1. **WiFi Direct (P2P)** - прямое соединение без интернета
   - ✅ Прямое соединение без сервера
   - ✅ Высокая скорость передачи
   - ❌ Ограничение по дальности (~10-20 метров)
   - ❌ Требует настройки соединения

2. **Локальный HTTP сервер на основном устройстве**
   - ✅ Простота реализации
   - ❌ Работает только в локальной сети
   - ❌ Нет глобального доступа

3. **Расширение существующей Yandex Cloud инфраструктуры** (РЕКОМЕНДУЕТСЯ)
   - ✅ Нет необходимости в дополнительном сервере
   - ✅ Работает через интернет (любое расстояние)
   - ✅ Уже реализована инфраструктура в приложении
   - ✅ Безопасность через OAuth

## Рекомендуемый подход: Расширение Yandex Cloud

### Архитектура решения:

```
┌─────────────────┐                    ┌─────────────────┐
│  Главное        │                    │  Приемник       │
│  приложение     │◄───────────────────┤  (другое        │
│  (охрана)       │    HTTP/HTTPS      │  устройство)    │
│                 │                    │                 │
│ - Отчеты →     │    Yandex Cloud    │ - Отчеты ←     │
│ - Статус →     │    Storage         │ - Статус ←     │
│ - Настройки →  │    Bucket          │ - Настройки ←  │
└─────────────────┘                    └─────────────────┘
```

### Что нужно реализовать:

#### 1. **Новое приложение "Ohrana Receiver"**
- Получение отчетов из Yandex Cloud Storage
- Просмотр статистики
- Удаленная настройка основного приложения

#### 2. **Обновление текущего приложения:**
- Выгрузка отчетов в облако (уже есть)
- Поддержка команд из облака (для удаленной настройки)

## Требования к реализации

1. **Безопасность:**
   - Использовать OAuth токены
   - Хранить токены в SharedPreferences
   - Шифровать чувствительные данные

2. **Оффлайн режим:**
   - Кэшировать последние отчеты локально
   - Работать без интернета (показывать кэш)

3. **Производительность:**
   - Асинхронная загрузка
   - Прогресс бар для больших файлов
   - Обработка ошибок сети

4. **Юзабилити:**
   - Уведомления о новых отчетах
   - Автоматическая синхронизация
   - Понятные сообщения об ошибках

## Ожидаемое поведение

### Для пользователя:

1. **Передача отчета:**
   - Охранник завершает смену → отчет загружается в облако
   - Приемник открывает приложение → видит список доступных отчетов
   - Нажимает на отчет → открывает в браузере/приложении

2. **Удаленная настройка:**
   - Администратор меняет настройки на основном устройстве
   - Настройки автоматически загружаются в облако
   - Приемник получает обновления и применяет их

### Для разработчика:

1. **Безопасность:** OAuth токены, шифрование
2. **Производительность:** Асинхронная загрузка, кэширование
3. **Масштабируемость:** Легко добавить новые типы данных

## Заключение

Создание отдельного приложения для приема сообщений и настройки возможно и реализуемо через существующую инфраструктуру Yandex Cloud. Это позволит:

- Передавать отчеты между устройствами через интернет
- Удаленно управлять настройками основного приложения
- Не требовать дополнительного сервера
- Использовать уже существующую аутентификацию

Рекомендуется начать с Phase 1 (базовая инфраструктура приемника), затем перейти к Phase 2 (удаленная настройка) и Phase 3 (интеграция).
