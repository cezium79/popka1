# План исправления ошибки File.io

## Проблема
После исправления `sequenceErrorType` и переписи на OkHttp, при отправке в File.io возникает ошибка:
```
Value<html> jf typeJava.lang.String cannjt be converted to JSONObject
```

Это означает, что API возвращает HTML вместо JSON, что указывает на неправильный endpoint или неправильный формат запроса.

## Найденные факты
1. Правильный endpoint для File.io API: `https://file.io`
2. API принимает файлы через `multipart/form-data` с ключом "file"
3. Требуется заголовок `Accept: application/json` для получения JSON ответа
4. Рабочая версия в файле `fileio.kt` использует тот же endpoint и метод

## Реальная причина ошибки
**ДВОЙНОЕ ЧТЕНИЕ response.body!** В коде мы читали `response.body?.string()` дважды:
- Первый раз в блоке `if (!response.isSuccessful)` для логирования ошибки
- Второй раз при `val responseBody = response.body?.string()` для парсинга JSON

При втором вызове `body?.string()` возвращает пустую строку или HTML (в зависимости от сервера), что вызывает ошибку `JSONException`.

## Решение
1. Вынести `responseBody` в переменную ОДИН раз после выполнения запроса
2. Сначала проверять `response.isSuccessful` ПЕРЕД парсингом JSON
3. Если код ответа не 200, не пытаться парсить JSON
4. Логировать полный ответ для отладки

## Файлы для изменения
- ✅ `CloudStorageManager.kt` - исправлено чтение response.body и добавлена проверка успешности запроса

## Изменения в коде
```kotlin
// Выполняем запрос
client.newCall(request).execute().use { response ->
    // Сначала считываем response body ОДИН раз!
    val responseBody = response.body?.string() ?: ""
    
    // Логируем полный ответ для отладки
    Log.i(TAG, "Response code: ${response.code}")
    Log.i(TAG, "Response body: $responseBody")
    
    // Сначала проверяем успешность запроса ПЕРЕД парсингом JSON!
    if (!response.isSuccessful) {
        Log.e(TAG, "Server error: ${response.code}")
        return Result.failure(Exception("Ошибка сервера File.io: ${response.code}"))
    }
    
    // Парсим JSON-ответ (только если код 200)
    val json = org.json.JSONObject(responseBody)
    // ... остальной код
}
```

# План перехода на OkHttp

## Проблема
В `CloudStorageManager.kt` используется низкоуровневый `HttpsURLConnection` для работы с Yandex Cloud и Yandex Disk API. Это усложняет код и требует ручного управления соединениями, заголовками и потоками.

## Методы, использующие HttpsURLConnection

### 1. uploadFileToCloud()
- **Файл**: `CloudStorageManager.kt` (строки ~124-185)
- **Цель**: Загрузка файлов в Yandex Cloud Storage
- **Текущий API**: `HttpsURLConnection` для PUT запроса

### 2. getUploadLinkForDisk()
- **Файл**: `CloudStorageManager.kt` (строки ~198-278)
- **Цель**: Получение ссылки для загрузки файла в Яндекс.Диск
- **Текущий API**: `HttpsURLConnection` для GET запроса

### 3. uploadFileToDisk()
- **Файл**: `CloudStorageManager.kt` (строки ~285-413)
- **Цель**: Загрузка файлов в Яндекс.Диск (с предварительным созданием директорий)
- **Текущий API**: `HttpsURLConnection` для GET и PUT запросов

### 4. checkDirectoryExists()
- **Файл**: `CloudStorageManager.kt` (строки ~420-451)
- **Цель**: Проверка существования директории в Яндекс.Диске
- **Текущий API**: `HttpsURLConnection` для GET запроса

### 5. createSingleDirectory()
- **Файл**: `CloudStorageManager.kt` (строки ~458-492)
- **Цель**: Создание директории в Яндекс.Диске
- **Текущий API**: `HttpsURLConnection` для PUT запроса

### 6. getPublicUrlForDisk()
- **Файл**: `CloudStorageManager.kt` (строки ~471-505)
- **Цель**: Получение публичного URL файла в Яндекс.Диске
- **Текущий API**: `HttpsURLConnection` для GET запроса

### 7. uploadHtmlToDiskDirect()
- **Файл**: `CloudStorageManager.kt` (строки ~1619-2018)
- **Цель**: Загрузка HTML отчета напрямую в Яндекс.Диск без сохранения на телефоне
- **Текущий API**: `HttpsURLConnection` для GET и PUT запросов

### 8. uploadHtmlToFileIoDirect()
- **Файл**: `CloudStorageManager.kt` (строки ~1289-1381)
- **Цель**: Загрузка HTML отчета напрямую в File.io без сохранения на телефоне
- **Текущий API**: `HttpURLConnection` для POST запроса с multipart/form-data

### 9. uploadHtmlToFileIoBinary()
- **Файл**: `CloudStorageManager.kt` (строки ~1383-1445)
- **Цель**: Альтернативная загрузка в File.io как binary data
- **Текущий API**: `HttpURLConnection` для POST запроса

## Решение
Переписать все методы для использования **OkHttp** вместо `HttpsURLConnection`/`HttpURLConnection`.

### Преимущества OkHttp:
- Автоматическое управление соединениями
- Простой и читаемый API
- Встроенная поддержка gzip, кэширования, перенаправлений
- Лучшая обработка ошибок
- Поддержка async/await (через coroutines)

### Шаги реализации:
1. Добавить импорт OkHttp (уже добавлен: `import okhttp3.*`)
2. Создать глобальный экземпляр `OkHttpClient` для повторного использования
3. Переписать каждый метод, используя `client.newCall(request).execute()`
4. Удалить импорт `javax.net.ssl.HttpsURLConnection` и связанные классы
5. Протестировать каждый метод
